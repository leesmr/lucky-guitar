ly lz ma mb mc md me mf mg mh"><strong class="lw mi">GitHub</strong> is a code hosting platform for version control and collaboration. It’s designed around Git, a system for tracking changes in software code. It lets you and others work together on projects from anywhere. In this article we’ll take a look at the key concepts of the GitHub development workflow. Including working with repositories, branches, forks, commits, pull requests and merging. Let’s get started!</p><h2 id="cf99" class="mj mk ef aq ap ed ml mm mn mo mp mq mr ms mt mu mv">What is version control?</h2><p id="e9de" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh">Version control is a system that helps developers track and manage changes to a software projects code. As projects grow, the need for version control becomes vital — especially in a collaborative project. We can work safely by using whats known as <strong class="lw mi">branching</strong> and <strong class="lw mi">merging</strong>.</p><p id="9ffe" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><strong class="lw mi">Branching</strong> allows us to duplicate our source code (aka ‘the repository’), so we can safely make changes without affecting the entire project. Once the changes have been reviewed and approval is agreed upon, we <strong class="lw mi">merge</strong> our branch into the master to update our official code. If any bugs are identified— we have the option to revert back to our original code, as our changes have been tracked.</p><h2 id="2214" class="mj mk ef aq ap ed ml mm mn mo mp mq mr ms mt mu mv">What is Git?</h2><p id="d109" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh">Git and Github are not synonymous! Git is a specific open-source version control system created by <a href="https://en.wikipedia.org/wiki/Linus_Torvalds" class="cz bv nb nc nd ne" target="_blank" rel="noopener">Linus Torvalds</a> in 2005. It’s the program that actually tracks your changes, and ensures the entire code-base is available on each developers computer. GitHub however, is simply hosting your repositories (as well as providing some additional functionality).</p><h1 id="c641" class="nf mk ef aq ap ed ng nh ni nj nk nl nm nn no np nq">The Fundamentals</h1><p id="3246" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh">Let’s get started! If you haven’t signed up for a GitHub account, go ahead and <a href="https://github.com/join" class="cz bv nb nc nd ne" target="_blank" rel="noopener">sign up here</a>.</p><h2 id="c93b" class="mj mk ef aq ap ed ml mm mn mo mp mq mr ms mt mu mv">Creating a Repository</h2><p id="4b9e" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh">A GitHub <strong class="lw mi">repository</strong> (or “repo”) can be thought of as the root folder for your project. It contains all your project files and gives you the ability to access each files revision history. If you’re working in a team you can give other people access to your repository for project collaboration.</p><p id="0b2a" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">Lets create our first repository! Make sure your signed in to GitHub, then:</p><ol class=""><li id="9e35" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh nr ns nt">In the upper-right corner, click the settings <code class="lp nu nv nw nx b">+</code> icon &amp; then New repository</li><li id="19dc" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Give your repository a name</li><li id="ce4e" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Add a description if you like</li><li id="4aad" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Choose between creating a public or private repository (either are fine)</li><li id="cf23" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Select Initialize this repository with a README (eg. “My first repo!”)</li><li id="8852" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click Create repository!</li><li id="1593" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Add files to your repository —we’re only testing so it really doesn&#x27;t matter what you add. Find an HTML file for example, and add it to your repository via the Upload files button.</li></ol><h2 id="2bae" class="mj mk ef aq ap ed ml mm mn mo mp mq mr ms mt mu mv">Creating an Issue</h2><p id="a556" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh"><strong class="lw mi">Issues</strong> are how we track the tasks, enhancements, and bugs in our projects. They’re meant to be shared amongst your team to facilitate discussion for review as well as for managing task delegation. If you open an issue on a project managed by someone else, it’ll stay open until either you close it (for example if you figure out the problem) or if the repo owner closes it. When you create an issue, be sure to give a clear explanation of the task at hand. Let’s create an issue in our repository:</p><ol class=""><li id="167d" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh nr ns nt">In your repository, select the Issues tab</li><li id="1d8d" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click the New Issue button</li><li id="3262" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Give your issue the title ‘Setup GitHub pages’</li><li id="b37e" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Give a clear description e.g. ‘Need to setup GitHub Pages in this repository’</li><li id="8c01" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click Submit new issue</li></ol><p id="d3a4" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><strong class="lw mi">Assigning an Issue</strong></p><p id="874a" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">We need to <strong class="lw mi">assign</strong> issues so our team members know who’s job it is to handle the task!</p><ul class=""><li id="ab9a" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh od ns nt">On the right side of the screen, under the “Assignees” section, click the settings icon and select yourself</li></ul><p id="0668" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">Let’s resolve our first issue! We want to <em class="oe">Setup GitHub pages</em>. You can read about GitHub Pages <a href="https://pages.github.com/" class="cz bv nb nc nd ne" target="_blank" rel="noopener">here</a>. But for now, our focus on completing the task:</p><ol class=""><li id="77d0" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh nr ns nt">Click on the Settings tab in your repository</li><li id="db90" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Scroll down to the “GitHub Pages” section</li><li id="9738" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">From the “Source” drop down, select master branch</li><li id="7477" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click Save</li></ol><p id="f4f2" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><strong class="lw mi">Closing an Issue</strong></p><p id="e272" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">Now you’ve completed the task — you can go ahead and close it! You can delete an issue on GitHub, however closing it tells your team members that the task has been completed. To close:</p><ul class=""><li id="95ce" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh od ns nt">Open up your completed issue and click Close issue</li></ul><h2 id="3915" class="mj mk ef aq ap ed ml mm mn mo mp mq mr ms mt mu mv"><strong class="bc">The GitHub Flow</strong></h2><p id="6c1f" class="lu lv ef aq lw b lx mw lz mx mb my md mz mf na mh">Now that we know how to work with issues, it’s time to look at the <strong class="lw mi">GitHub Flow</strong>. Simply put its a workflow where we can experiment with new ideas safely, without the risk of compromising our project. This is primarily achieved through the use of <strong class="lw mi">branching</strong>.</p><figure class="ld le lf lg lh fj dl dm paragraph-image"><div class="li lj et lk af"><div class="dl dm of"><div class="lo l et lp"><div class="og l"><div class="dp ll gq n o gp af ca lm ln"><img alt="" class="gq n o gp af lr ls lt" src="https://miro.medium.com/max/60/1*4Og3D50CV9naY0oVowyeSw.png?q=20" width="800" height="201" role="presentation"/></div><img alt="" class="dp ll gq n o gp af gn" width="800" height="201" role="presentation"/><noscript><img alt="" class="gq n o gp af" src="https://miro.medium.com/max/1600/1*4Og3D50CV9naY0oVowyeSw.png" width="800" height="201" role="presentation"/></noscript></div></div></div></div><figcaption class="au ei oh oi hv dn dl dm oj ok ap cu"><a href="https://guides.github.com/introduction/flow/" class="cz bv nb nc nd ne" target="_blank" rel="noopener">GitHub Flow</a></figcaption></figure><p id="ad3b" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">By default, our project lives on the master branch — any changes to the master will update directly to our project (This can be dangerous is you haven’t properly reviewed your changes!).</p><p id="4eae" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">When we want to experiment with a new feature, or even fix an issue, we create a new <strong class="lw mi">branch</strong> on the project. The branch will initially be a duplicate of your master, now when you make changes —they’ll reflect only on the branch.</p><p id="9535" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">While working on changes, you’ll<strong class="lw mi"> commit</strong> the changes to your branch. When you’re satisfied that the changes are complete, its time to open a <strong class="lw mi">pull request.</strong> From here your team will discuss and further refine the project changes. Once the changes have been approved, the branch will be <strong class="lw mi">merged</strong> onto the <strong class="lw mi">master branch</strong>. Let’s go through an example of this process now!</p><p id="32c9" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><strong class="lw mi">Creating a Branch</strong></p><ol class=""><li id="6370" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh nr ns nt">Open your repository and click the Code tab</li><li id="91eb" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click <code class="lp nu nv nw nx b">Branch: master</code> in the drop-down</li><li id="d82f" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">In the field, enter a name for your branch (e.g. ‘development’)</li><li id="367a" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click Create branch</li></ol><p id="b722" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">Now that you’ve created a branch, you can modify your project without changing the deployed <code class="lp nu nv nw nx b">master</code> branch.</p><p id="d666" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><em class="oe">A note on forks —</em> A <strong class="lw mi">fork</strong> is different to a branch in that it allows you to <strong class="lw mi">clone</strong> another repo in your own account. It essentially allows you to start a new project based on a previous project.</p><p id="069c" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh"><strong class="lw mi">Committing a file</strong></p><p id="8eba" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">Now we can safely work within our branch, let’s create a file and make our first commit.</p><ol class=""><li id="d51f" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh nr ns nt">Make an edit (anything! just add a simple comment to your code) to one of the files you had added to the repository earlier</li><li id="26c2" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Give the commit a name and description</li><li id="338f" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Make sure your newly created (development) branch is selected</li><li id="ce44" class="lu lv ef aq lw b lx ny lz nz mb oa md ob mf oc mh nr ns nt">Click Commit changes</li></ol><p id="b3d6" class="lu lv ef aq lw b lx ly lz ma mb mc md me mf mg mh">You’ve made your first commit!
